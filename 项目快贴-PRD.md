# PRD：项目快贴（Mac 个人本地版）

## 1. 产品定位

项目快贴是一个面向重度 AI 用户和开发者的 Mac 本地快捷输入工具。

用户在常见文本输入场景中工作时，通过全局快捷键呼出一个轻量面板，快速检索并执行当前项目常用的 Prompt、代码片段和说明模板，将内容以“自动粘贴优先，剪贴板兜底”的方式送入当前输入上下文。

这个产品第一版只解决一个核心问题：

**把当前项目里高频重复输入的内容，用尽可能低的操作成本送进当前输入框。**

---

## 2. 目标用户

核心用户只包括两类：

- 重度 AI 用户：高频使用 ChatGPT、Claude、Cursor、Copilot 等工具，反复输入 Prompt、任务说明、格式约束和上下文模板的人
- 开发者：需要频繁插入代码片段、命令、提交说明、接口描述、排障模板和项目上下文的人

这不是一个面向泛用文本记录或轻办公场景的工具。

---

## 3. 核心使用场景

第一版聚焦以下高频场景：

- 在 AI 对话工具中快速插入 Prompt 模板、角色设定、输出格式要求、上下文说明
- 在编辑器或终端相关输入场景中快速插入代码片段、命令、提交说明、排障模板
- 在浏览器、文档或 IM 工具中快速插入当前项目常用说明、状态回复、交付说明
- 在多个项目之间切换时，快速调用不同项目下的常用内容

典型流程：

1. 用户正在支持的常见应用中工作
2. 按下全局快捷键
3. 面板弹出并聚焦搜索
4. 用户搜索或直接选择某个词条
5. 系统先写入剪贴板，再尝试自动粘贴到当前输入位置
6. 成功后关闭面板，用户继续工作
7. 如果自动粘贴失败，用户至少可以立即手动粘贴

---

## 4. 产品边界

### 4.1 第一版要做的

- 全局快捷键呼出面板
- 基于项目组织和检索词条
- 支持键盘优先的快速搜索和执行
- 词条执行时先复制，再自动粘贴
- 自动粘贴失败时保证剪贴板兜底
- 提供基础的项目管理和词条管理

### 4.2 第一版不做的

- 团队协作
- 云同步
- 剪贴板历史管理
- 知识库或文档系统
- 复杂工作流编排
- 轻量待办、提醒、任务管理
- 自动根据 IDE、目录或前台窗口识别项目
- 模板变量编辑器、表单式参数填充

原则很简单：第一版只服务主链路，不做与“快速调用并输入项目常用内容”无关的延伸需求。

---

## 5. 核心对象定义

### 5.1 项目

项目是词条的一级归属单位，用来隔离不同工作上下文。

每个项目至少包含：

- 名称
- 创建时间
- 更新时间

系统必须内置一个不可删除的 `通用项目`，用于存放跨项目复用内容。

### 5.2 词条

词条是用户可被快速执行的内容单元。虽然对外仍可叫“词条”，但设计和实现上应按“片段 / 模板”对待，而不是只按短文本对待。

每条词条第一版至少包含：

- 标题
- 内容
- 所属项目

第一版必须包含以下字段，用于排序和重用：

- 类型：`prompt` / `code` / `reply` / `note`
- 是否置顶
- 排序值
- 最近使用时间
- 使用次数

说明：

- 词条内容必须支持多行文本
- 词条内容不限制长度，但面板中只展示预览
- `type` 字段第一版仅用于数据标识和后续扩展预留，不进入筛选、排序和复杂交互
- 第一版不做变量占位和动态参数填充

---

## 6. 核心需求

### 6.1 全局快捷键呼出面板

目标：在常见文本输入场景中，都能快速调出项目快贴面板。

要求：

- 支持设置一个全局快捷键
- 按一次弹出面板
- 面板已打开时，再按一次可关闭
- 按 `Esc` 可关闭
- 点击面板外区域可关闭
- 面板弹出后默认聚焦搜索框
- 面板关闭后不抢占用户后续操作

适用范围说明：

- 第一版面向常见文本输入场景，优先支持 AI 对话工具、编辑器、浏览器和 IM 工具
- 第一版不承诺所有应用都能稳定完成自动粘贴
- 对兼容性不稳定的应用，至少保证成功复制到剪贴板

### 6.2 面板展示与交互

目标：让用户用最少操作找到并执行目标词条。

要求：

- 默认展示当前项目词条
- 默认同时展示 `通用项目` 词条
- 每个词条至少展示标题和内容预览
- 支持滚动查看
- 支持鼠标点击执行
- 支持键盘上下选择
- 支持回车执行
- 支持搜索过滤
- 支持纯键盘完成“呼出 -> 搜索 -> 选择 -> 执行”

展示要求：

- 当前项目词条和 `通用项目` 词条默认混排展示
- 来自 `通用项目` 的词条应有轻量来源标识，避免用户误判上下文
- 快捷面板内必须支持切换当前项目
- 项目切换应是面板内的一等操作，不能要求用户必须进入主界面才能切换
- 切换项目后，面板结果应立即刷新，用户可继续在当前面板内检索和执行

搜索要求：

- 支持按标题搜索
- 支持按内容搜索
- 搜索结果应即时更新，不依赖显式提交

排序要求：

- 置顶词条优先
- 置顶之后，按手动排序值排序
- 若排序值相同，按最近使用时间排序
- 若最近使用时间相同，再按使用次数排序

说明：

- 数字快捷选择不是第一版必需项，有余力再做
- 第一版优先保证搜索速度和键盘路径顺畅，不追求复杂视觉结构

### 6.3 词条执行行为

目标：选择词条后，内容尽可能直接进入当前输入框，而不是只停留在复制层面。

执行顺序必须是：

1. 将词条内容写入剪贴板
2. 尝试向当前输入位置发送粘贴操作
3. 成功后关闭面板

失败兜底要求：

- 如果自动粘贴失败，剪贴板中仍必须保留目标内容
- 用户应能立即手动粘贴
- 系统应给出轻量但明确的失败提示，不允许静默失败
- 自动粘贴失败时，面板应关闭并让用户返回原工作流，再提示“已复制，可手动粘贴”

行为约束：

- 自动输入优先，复制是兜底
- 第一版不要求对所有应用实现 100% 自动输入成功
- 第一版不做“发送后自动按回车”
- 第一版默认不恢复用户原剪贴板内容，优先保证执行确定性和实现稳定性

### 6.4 项目管理

目标：让用户能明确区分不同项目上下文，避免词条混用。

支持操作：

- 新建项目
- 重命名项目
- 删除项目
- 切换当前项目

约束：

- `通用项目` 不可删除
- 删除非空项目时，不允许直接删除，必须先迁移或删除其下全部词条
- 当前项目切换后，面板展示内容要立即刷新
- 删除非空项目时，应要求用户先选择迁移目标项目，迁移完成后才允许删除

第一版项目切换原则：

- 只做手动切换
- 快捷面板内支持快速切换项目
- 不做自动识别当前代码仓库、目录或前台应用
- 不做根据编辑器上下文自动推断当前项目

### 6.5 词条管理

目标：用户可以低成本维护自己的常用内容库，并立刻用于面板执行。

支持操作：

- 新增词条
- 编辑词条
- 删除词条
- 搜索词条
- 设置所属项目
- 置顶或调整排序

约束：

- 保存后应立即可在面板中调用
- 编辑时必须支持多行内容输入
- 删除操作应有明确确认，避免误删高频内容

### 6.6 主界面

目标：承载维护操作，不把复杂管理动作塞进快捷面板。

第一版主界面要求：

- 第一版必须提供独立主界面窗口
- 主界面用于项目管理、词条管理和基础设置
- 用户应能从应用常驻入口进入主界面
- 快捷面板只负责快速检索和执行，不承担复杂编辑任务

信息架构原则：

- 快捷面板是执行入口
- 主界面是维护入口
- 两者职责必须清晰分离

---

## 7. 平台约束与兼容性要求

这些不是实现细节，而是第一版必须正视的产品约束。

### 7.1 权限要求

要实现全局快捷键和跨应用自动粘贴，产品需要依赖 macOS 相关权限能力。

第一版必须提供：

- 首次启动的权限引导
- 权限缺失时的明确提示
- 权限未开启时的可降级行为说明

权限降级要求：

- 缺少自动粘贴所需权限时，用户仍应可以呼出面板、搜索词条和复制内容
- 缺少相关权限时，系统不执行自动粘贴，并明确提示当前处于“仅复制模式”
- 用户完成权限配置后，产品应可恢复自动粘贴能力

### 7.2 兼容性原则

不同应用对自动粘贴、焦点恢复和快捷键响应的兼容性不同。

第一版兼容性原则：

- 对常见文本输入场景优先保证可用
- 对无法稳定自动粘贴的应用，至少保证“成功复制到剪贴板”
- 产品承诺的是“自动粘贴优先 + 剪贴板兜底”，不是“所有应用无差别强保证”

### 7.3 稳定性要求

第一版必须避免以下情况：

- 面板弹出后无法关闭
- 执行后既没有自动粘贴也没有成功复制
- 当前项目切换后仍展示旧项目数据
- 删除项目或词条后状态不同步

---

## 8. 主流程

### 8.1 快速输入主流程

1. 用户在支持的常见应用中输入
2. 按下全局快捷键
3. 面板弹出并聚焦搜索框
4. 用户搜索或直接选择词条
5. 系统写入剪贴板
6. 系统尝试自动粘贴到当前输入位置
7. 若成功，关闭面板并返回工作流
8. 若失败，关闭面板，保留剪贴板内容，并提示用户手动粘贴

### 8.2 项目切换流程

1. 用户打开面板或主界面
2. 切换当前项目
3. 系统立即刷新当前项目词条
4. 后续呼出的面板按新项目展示内容

### 8.3 词条维护流程

1. 用户进入主界面
2. 新增、编辑或删除词条
3. 保存后数据立即生效
4. 面板中可直接检索和执行最新内容

---

## 9. 第一版必须做对的事

### 9.1 主链路足够顺

最重要的链路只有一条：

**按键 -> 弹面板 -> 搜索 / 选择 -> 执行 -> 内容进入当前工作流**

如果这条链路不够快、不够稳，产品就不成立。

### 9.2 键盘路径必须完整

重度 AI 用户和开发者不会接受必须频繁切鼠标的面板。

第一版至少要保证：

- 呼出后可直接输入搜索
- 可用方向键选择结果
- 可用回车完成执行
- 整个过程可以不依赖鼠标

### 9.3 自动粘贴必须有兜底

自动粘贴不是绝对可靠能力，因此必须保证：

- 先复制，再尝试粘贴
- 失败时内容仍在剪贴板
- 用户知道下一步该怎么做

### 9.4 项目隔离必须清晰

项目是这个工具的上下文核心。

第一版必须避免：

- 词条归属混乱
- 当前项目不明确
- 项目切换后展示与执行不一致

---

## 10. 第一版验收标准

满足以下条件，才算第一版达到可用标准：

- 用户可以在常见文本输入场景中通过全局快捷键稳定呼出面板
- 用户可以纯键盘完成搜索、选择和执行
- 词条执行时一定先写入剪贴板
- 自动粘贴失败时有明确提示，且用户可以立即手动粘贴
- 项目和词条的增删改查闭环完整
- 当前项目与 `通用项目` 的展示逻辑一致且稳定
- 删除非空项目时不会发生词条静默丢失
- 快捷面板与主界面职责清晰，不互相混用

最低性能验收标准：

- 面板从触发快捷键到可输入搜索的目标时间不超过 300ms
- 搜索结果刷新目标时间不超过 100ms
- 常规词条执行从确认到完成复制的目标时间不超过 200ms
- 权限缺失时，用户应能完成引导配置，或明确走复制兜底路径

内部首批验收清单：

- ChatGPT Web
- Claude Web
- Cursor
- VS Code
- Chrome 常见表单输入框
- 常见 IM 输入框中的基础文本输入场景

如果以上几点没有做好，不应继续扩展边缘功能。

---

## 11. 后续版本再考虑的方向

这些方向有价值，但不进入第一版：

- 变量模板和参数化填充
- 自动识别当前目录、仓库或 IDE 项目
- 快速重复执行上一条词条
- 导入导出
- 多设备同步
- 更细粒度标签、过滤器和高级排序

第一版先把“高频项目内容快速注入当前输入上下文”做扎实，再决定是否扩展。
