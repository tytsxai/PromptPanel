# PromptPanel 关键模块与核心逻辑

更新时间：`2026-05-07`

## 1. 模块分层总览

PromptPanel 当前是典型的“仓库 + 服务 + 视图模型 + 视图”结构，但真正关键的是系统集成层和执行编排层。

| 层级 | 代表文件 / 目录 | 主要职责 |
| --- | --- | --- |
| 应用装配层 | `App/AppDelegate.swift` | 初始化数据库、仓库、服务、窗口和回调 |
| 共享状态层 | `App/AppState.swift` | 当前项目、面板显隐、权限状态、面板固定状态 |
| 数据层 | `Core/Database`、`Core/Repositories` | SQLite 打开、迁移、查询、写入 |
| 业务服务层 | `Core/Services` | 搜索、执行、面板、权限、存储维护 |
| 系统集成层 | `Integrations/*` | 系统 API 和第三方库的薄封装 |
| 视图模型层 | `Features/*ViewModel.swift` | 页面状态、用户动作、与服务/仓库交互 |
| 视图层 | `Features/*View.swift` | 主窗口与快捷面板 UI |
| 发布交付层 | `scripts/`、`.github/workflows` | 打包、公证、恢复、CI 验证 |

## 2. 启动与装配模块

### 2.1 `PromptPanelApp`

- 只是 SwiftUI 的壳入口
- 真实控制权在 `AppDelegate`

### 2.2 `AppDelegate`

这是整个项目的装配中心，负责：

- 启动时做同 bundle id 单实例保护，避免多个实例同时抢热键、写库和处理焦点
- 初始化 `DatabaseManager`
- 初始化 `ProjectRepository / EntryRepository / SettingsRepository / LogRepository`
- 初始化权限、剪贴板、自动粘贴、登录启动、更新、存储维护等服务
- 组装 `QuickPanelViewModel` 和 `MainWindowViewModel`
- 注册托盘、快捷键和窗口回调
- 启动后触发存储维护、更新器、权限刷新和异常提示
- 根据 QA 环境变量自动打开主窗口或快捷面板，供 Computer Use / UI QA 直接抓取可见窗口

如果你新增了新的跨模块能力，优先在这里完成依赖装配，而不是让 View 自己 new service。

### 2.3 `AppLaunchCoordinator`

负责单实例保护：

- 默认检测相同 bundle id 下是否已有运行实例
- 如果已有实例，新启动的进程会尝试激活旧实例后退出
- `PROMPTPANEL_ALLOW_EXISTING_INSTANCE=1` 可跳过检查，仅用于隔离 QA、release smoke 或临时并行验证

## 3. 数据模块

### 3.1 `DatabaseManager`

负责数据库生命周期：

- 创建并打开 SQLite
- 打开前确保目录权限正确
- 开启 `WAL`
- 注册并执行全部 migration
- 在库文件损坏时隔离旧库并重建
- 在 migration 失败时保留原库并报错退出

这是数据可靠性的关键边界，不能随便“为了方便调试”改成启动失败就自动擦库。

### 3.2 `Migrations`

当前 schema 演化顺序：

- `v1_create_tables`
- `v2_execution_log_diagnostics`
- `v3_execution_log_interaction_diagnostics`
- `v4_entry_tags`：`entries` 新增 `tags TEXT NOT NULL DEFAULT '[]'` 列；内容为 JSON 字符串数组（例：`["发布","检查清单"]`）。JSON 非法时解码回空数组，不影响主链路。
- `v5_drop_unused_entry_tags_index`：删除历史遗留 `index_entries_on_tags`，避免维护不存在的标签索引假设。

新增字段、索引或表时，必须加新 migration，不要回头改旧 migration 当成“当前真相”。

### 3.3 Repositories

#### `ProjectRepository`

负责：

- 项目列表
- 项目列表必须保持“默认项目优先，其余按名称排序”；GRDB 多字段排序要放在同一次 `order(...)` 调用里，连续调用会覆盖前一个排序条件
- 默认项目读取
- 项目创建、重命名、删除
- 非空项目迁移后删除

#### `EntryRepository`

负责：

- 词条列表与 FTS 搜索
- 当前项目 + 默认项目混排
- 使用次数和最近使用时间统计
- 固定排序规则
- 混排查询、普通列表和 FTS 查询都必须保留最终稳定排序键，避免同权重词条在刷新时随机跳动
- 词条 `tags` 字段以 JSON 字符串形式持久化；`Entry.init(row:)` 与 `encode(to:)` 在读写时通过 `EntryTagsCodec` 做 `[String] <-> JSON` 转换
- FTS 目前只覆盖 `title + content`，标签筛选在服务/视图模型层（`QuickPanelViewModel` 用 `#tag` 语法，`MainWindowViewModel.displayedEntries` 用 `entryTagFilter`）完成

当前排序的核心原则是：

1. 置顶优先
2. `sortOrder` 高的优先
3. 最近使用优先
4. 使用次数高的优先
5. 当前项目优先于默认项目
6. 完全同权重时按 `updatedAt DESC, id ASC` 收口，保证刷新、筛选和快捷编号不漂移

#### `SettingsRepository`

负责轻量 KV 配置：

- 当前项目
- 面板固定状态
- 面板内容尺寸
- 面板窗口位置
- 面板底部提示开关
- 面板紧凑行开关
- 应用主题
- 内容库排序模式

#### `LogRepository`

负责执行日志：

- 写入执行结果
- 拉取最近记录
- 汇总健康统计
- 清理过旧日志

## 4. 面板与焦点链路

### 4.1 `HotkeyService`

职责很单纯：

- 注册全局快捷键
- 迁移历史默认热键到 `⌥2`
- 在快捷键触发时调用 `PanelService`

它不直接处理窗口、搜索或执行逻辑。

### 4.2 `PanelService`

这是快捷面板稳定性的核心模块，负责：

- 记录呼出前的目标前台应用
- 创建 `NSPanel`
- 切换应用激活策略
- 把面板带到最前
- 稳定化激活和 key window 状态
- 隐藏时尽量恢复原前台应用
- 根据“固定置顶”状态决定是否在失焦时自动收起
- 根据设置页输入调整面板内容尺寸，并把窗口位置写回 `app_settings`
- 在固定面板长期可见时跟踪最近激活过的非 PromptPanel 应用，作为执行时的目标应用

这一层的复杂度来自 macOS 桌面行为本身，而不是业务功能。

### 4.3 `PanelOpenTracker`

负责面板时序诊断，例如：

- 热键触发时间
- 面板显示时间
- 搜索框真正拿到焦点时间
- 是否超过目标耗时

它不是业务逻辑，但对“为什么面板看起来弹了却不能马上输入”的排障很关键。

## 5. 搜索与面板状态模块

### 5.1 `QuickPanelViewModel`

负责快捷面板的运行时状态：

- 当前查询词
- 当前项目
- 搜索结果列表
- 当前选中项
- 权限状态提示
- 执行门禁

关键点有两个：

1. 搜索做了 debounce，避免每次输入都直接打数据库。
2. 面板刚刚获得焦点时存在短暂执行门禁，避免“只按热键就误触发回车执行”。

当前与视图层的接口边界也更明确：

- 正常态不主动抬出状态条；只有权限未授权或搜索/执行失败才向视图层暴露异常提示
- 搜索结果始终按“标题优先、内容预览次级、右侧元信息最轻”的紧凑行结构渲染
- 头部只保留项目切换、搜索和设置三个一等操作，不再往快捷面板里扩散维护型入口

### 5.2 `EntrySearchService`

负责把面板的搜索意图翻译成可执行查询：

- 当前项目 == 默认项目时，走单项目逻辑
- 空查询时，走浏览逻辑
- 有查询时，走 FTS 混合搜索逻辑
- 记录查询耗时和慢样本

这层故意独立出来，是为了让排序和搜索规则集中管理，不散落到 ViewModel 和 Repository 里。

## 6. 执行模块

### 6.1 `ExecuteService`

这是项目最关键的业务模块。

当前执行逻辑固定为：

1. 获取执行前前台应用 bundle id
2. 写入剪贴板
3. 关闭面板
4. 等待原目标应用恢复前台
5. 刷新辅助功能权限状态
6. 判断是否允许自动粘贴
7. 等待目标 App 恢复并短暂稳定焦点
8. 尝试发送 `Cmd+V`
9. 记录日志
10. 更新使用统计
11. 按结果决定是否提示用户

### 6.2 失败分类

当前失败原因被明确分类为：

- `clipboard_write_failed`
- `accessibility_not_granted`
- `target_app_not_restored`
- `paste_event_creation_failed`

这使得“为什么这次没贴上去”可以从执行日志直接缩小范围，而不是统一叫“失败”。

### 6.3 为什么先复制再粘贴

这是当前项目非常重要的设计原则：

- 剪贴板写入是最低保底
- 自动粘贴只是增强
- 即使自动粘贴失败，也不能让用户内容丢失

## 7. 系统集成模块

### `ClipboardService`

- 负责字符串写入剪贴板

### `PasteService`

- 负责构造并派发 `Cmd+V` 的 `CGEvent`
- 只关心“能不能派发”，不负责判断业务成功

### `PermissionService`

- 检查辅助功能权限
- 触发系统授权引导
- 打开系统设置对应页面

### `LoginItemService`

- 用 `SMAppService.mainApp` 注册 / 取消登录启动

### `UpdaterService`

- 解析 Sparkle 配置
- 当 feed URL 和公钥同时存在时启用更新检查
- 暴露当前更新状态给主界面

### `TrayManager`

- 提供菜单栏入口
- 从常驻态拉起主窗口或退出应用

## 8. 主窗口管理模块

### 8.1 `MainWindowViewModel`

负责所有维护型操作：

- 项目 CRUD
- 词条 CRUD
- 当前项目切换
- 权限状态查看与引导
- 登录启动切换
- 运行健康与执行日志展示
- 手动备份
- 更新检查

内容库预览必须跟随当前可见筛选结果：当类型/标签筛选把原选中词条过滤掉时，预览区应回落到当前可见列表第一条，而不是继续展示已被过滤掉的旧词条。

内容库本地排序也必须使用稳定兜底键：使用次数、最近时间或标题相同的时候，继续按更新时间/标题/id 收口，避免筛选、刷新或重载后选择状态漂移。

内容库可见列表由 `MainWindowViewModel.displayedEntries` 派生缓存承载。`entries` 加载完成、类型/标签筛选变化、排序模式变化时刷新一次缓存，视图层只读取缓存，避免计数、列表、选中态和预览区在同一次重绘里反复过滤排序。

### 8.2 `MainWindowView` / `LibraryView` / `SettingsView`

主窗口 2026-04-24 后的视图组织：

- `MainWindowView`：外壳。顶部自定义标题栏（预留红绿灯 + 居中 `内容库 / 设置` 分段控件），承载所有编辑型 sheet（项目 / 词条 / 项目迁移删除）。
- `LibraryView`：三栏内容库。
  - 左：项目侧栏（`全部项目 + 各项目`，含「当前」胶囊与计数；底部只保留“设为当前项目”这个强相关动作，重命名/删除继续走右键菜单）
  - 中：搜索框、过滤芯片（全部 / 类型 / 高频标签 Top 8）、排序菜单、新建按钮、词条列表（标题 / 预览 / 使用次数 / 最近时间 / 项目 / 内联标签）
  - 右：预览面板。类型胶囊、置顶胶囊、标题、主动作（复制 / 编辑 / 置顶 / 删除）、正文（`textSelection(.enabled)`）、底部元信息（项目 / 标签 / 更新时间）
- `SettingsView`：顶部健康摘要 + 左右双列 + 底部全宽详情
  - 顶部：正常态只显示一行安静摘要；权限未授权、没有备份或近 7 天执行失败才升成明确提示
  - 左列：外观、快捷键、面板行为、权限与启动
  - 右列：运行概况、维护操作
  - 底部：数据位置、最近执行记录

当前主窗口页面已经收成“弱卡片感 + 强分区感”的统一风格，后续继续扩展界面时，应优先复用现有的 section、row、badge、quiet status 这些轻量原语，而不是重新引入更重的仪表盘式承载层。

### 8.3 `Features/Shared/DesignComponents.swift`

对设计稿 token/primitive 的最小对齐：

- `KbdLabel`：键位胶囊
- `FilterChip`：通用筛选芯片（含类型和标签两种用法）
- `SectionHeading`：大写稀疏小标题
- `PrimaryActionButton` / `GhostActionButton` / `QuietIconButton`：预览面板主动作样式

Palette、surface、accent 等直接由 `Constants.VisualStyle` 统一管理，与 `frontend-draft/components/tokens.jsx` 字段一一对应。

### 8.4 `QuickPanelView` / `QuickPanelViewModel`

- 行布局对齐 `frontend-draft/components/quick-panel.jsx`：序号 + 类型图标 + 响应式标题 + 单行预览 + 使用次数 / 通用徽标 / 类型标签
- 面板列表密度：默认行高 30，紧凑行高 26；高频使用时优先减少上下留白，而不是放大单行展示
- 搜索入口由 SwiftUI 显式渲染 placeholder，不依赖透明 `NSSearchField` 的系统占位绘制；浅色 / 深色下都必须能看到“搜索项目 + #标签筛选”提示
- 面板位置：首次打开默认在当前屏幕中心偏左下；窗口移动后持久化 `panel_window_origin_x/y`，后续呼出优先恢复上次位置，并在屏幕变化时夹到可见区域内
- 搜索框支持 `#tag` 语法：`QuickPanelViewModel.extractTagFilter(from:)` 把 `#xxx` token 剥离出去，FTS 搜完再对结果做 `$0.tags.contains(tag)` 二次过滤
- 键盘直达：
  - `⌘1..⌘9` → `executeEntry(atNumber:)`，绕过执行就绪门禁（用户主动动作算 force execute）
  - `⌘C` → `copySelectionOnly()`，仅复制不粘贴并关闭面板；在搜索框有选区时让给系统默认
  - `↑↓` / `⏎` / `Esc` 沿用原路径
- 鼠标路径：整行都是命中区域，单击任意位置即执行可见词条并进入自动粘贴 / 复制兜底链路
- 执行节奏：不做逐字模拟输入；先写剪贴板，等待最近激活过的非 PromptPanel 目标 App 恢复焦点并短暂稳定后派发一次 `⌘V`，在速度和稳定性之间保持可控
- 底部键位提示同步更新为 `↑↓ / Enter / Esc / ⌘C / ⌘1-9 / ⌘P`
- 状态条只在异常时展示，正常态保持安静；项目切换、搜索、设置是头部仅保留的三个一等操作

## 9. 存储维护与恢复模块

### 9.1 `StorageMaintenanceService`

负责：

- 启动维护
- WAL checkpoint
- 自动备份
- 手动备份
- 备份清理
- 健康快照
- 打开数据目录与备份目录

### 9.2 恢复脚本 `restore-backup.sh`

恢复前会做：

- SQLite integrity check
- 必要表结构校验
- 默认项目校验
- 当前项目设置校验
- 运行中进程校验

然后才会：

- 把当前现场移到 `Recovery/manual-restore-时间戳/`
- 用目标备份恢复 `promptpanel.db`

## 10. 发布与验证模块

### 10.1 `build-app.sh`

负责把 SwiftPM 构建产物组装成真正可分发的 `.app`：

- 复制二进制
- 复制资源 bundle
- 嵌入 `Sparkle.framework`
- 补 rpath
- 注入版本和 Sparkle 配置
- 拒绝半套 Sparkle 配置：feed URL 和 EdDSA 公钥必须成对提供
- 签名前清理产物扩展属性，避免 framework 上的 FinderInfo / resource fork 污染使严格签名校验失败
- 做 ad-hoc 或正式签名
- 可选生成 zip

### 10.2 `release-readiness.sh`

这是发布前一站式自检入口，优先级很高。它把“脚本对不对、包能不能建、签名是不是坏的、隔离启动能不能起来”统一到一个命令里。打包后会按本次 `.app` 的版本号和 build 号精确定位 zip，避免输出目录残留历史归档时验错产物。

### 10.3 GitHub Actions

`.github/workflows/macos-release-readiness.yml` 当前固定执行：

- 脚本语法检查
- 非法备份恢复拒绝测试
- `swift build`
- `swift test`
- release 打包
- release bundle / zip 签名校验

## 11. 扩展时的最小接入方式

如果未来要新增功能，推荐按下面路径接入：

1. 明确新能力属于数据层、系统层还是业务流程层
2. 先补模型 / migration / repository
3. 再补 service
4. 再接入 ViewModel
5. 最后接 UI 和回归文档

不要直接跳到 View 里临时拼逻辑，否则这个项目很快会失去可维护性。
