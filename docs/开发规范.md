# PromptPanel 开发规范

更新时间：`2026-05-07`

本文把本仓库的日常开发、代码组织、测试、文档同步和 PR 要求整理成可执行规则。公开贡献入口见 [CONTRIBUTING](../CONTRIBUTING.md)。

## 1. 本地环境

最低要求：

- macOS 14 Sonoma 或更新版本。
- Xcode 15+，包含 macOS 14 SDK。
- Swift 5.10。
- 发布前需要完整 Xcode 提供的 `xctest` runner。

快速确认：

```bash
swift --version
xcode-select -p
xcrun --find xctest
swift build
./scripts/check-docs.sh
```

如果 `xcrun --find xctest` 失败，当前机器只能做构建和测试包编译，不能声明完整 XCTest 已执行。

## 2. 代码组织

| 层 | 路径 | 规则 |
| --- | --- | --- |
| 应用装配 | `Sources/PromptPanel/App` | 依赖组装、生命周期、单实例、启动维护。 |
| 数据模型 | `Sources/PromptPanel/Core/Models` | 只表达持久化和业务模型，不放 UI 状态。 |
| 数据库 | `Sources/PromptPanel/Core/Database` | migration 必须显式、可追踪、可测试。 |
| 仓库 | `Sources/PromptPanel/Core/Repositories` | 数据读写边界，避免 View 直接碰数据库。 |
| 服务 | `Sources/PromptPanel/Core/Services` | 业务编排和跨集成流程。 |
| 系统集成 | `Sources/PromptPanel/Integrations` | Clipboard、CGEvent、Hotkey、Tray、Sparkle 等薄封装。 |
| 面板功能 | `Sources/PromptPanel/Features/Panel` | 快捷面板 UI 和 ViewModel。 |
| 主窗口 | `Sources/PromptPanel/Features/MainWindow` | 内容库、设置、运行健康和维护入口。 |
| 共享 UI | `Sources/PromptPanel/Features/Shared` | 设计组件、键盘感知控件。 |
| UI 基准 | `frontend-draft/` | SwiftUI/AppKit 可见改动前必须先对齐这里。 |

不要新增顶层目录，除非它能明显降低长期维护成本并已在 issue 或 PR 中解释清楚。

## 3. Swift 编码规则

- 优先沿用现有 `final class`、`struct`、Repository、Service、ViewModel 分层。
- 业务逻辑不要塞进 SwiftUI View。
- 能通过依赖注入传入的能力，不新增全局单例。
- 新增持久化字段必须同步 migration、模型、仓库、测试和文档。
- 新增系统权限或 macOS API 接入时，先在 `Integrations/` 做薄封装。
- `ExecutionLog` 不能记录词条正文或用户私密 Prompt。
- 对复杂或风险边界明确的函数，用简短 `///` 注释说明契约；不要写重复代码字面的注释。

## 4. UI 规则

UI 改动顺序：

1. 更新 `frontend-draft/` 对应 JSX / HTML 基准。
2. 如果是视觉变化，更新或补充 `frontend-draft/uploads/` 截图。
3. 再修改 SwiftUI/AppKit。
4. 使用真实 `.app` 做至少一轮可见 smoke，不只看 `swift build`。

快捷面板的不可破坏约束：

- 搜索框打开后必须聚焦。
- `Esc` 关闭，方向键选择，回车执行。
- `Command-C` 是仅复制。
- `Command-P` 是面板固定切换。
- `Command-1..9` 是可见列表直达执行。
- 固定面板场景不能把目标 app 错误识别成 PromptPanel 自己。

## 5. 测试规则

普通改动：

```bash
swift build
swift test
./scripts/check-docs.sh
```

本机缺少 `xctest` 时：

```bash
swift build
swift build --build-tests
./scripts/check-docs.sh
./scripts/release-readiness.sh --allow-build-only-tests --skip-smoke-launch --output-dir /tmp/promptpanel-ready-check
```

发布前不允许用 build-only 结果替代完整测试：

```bash
./scripts/release-readiness.sh --public-distribution --sign-identity "Developer ID Application: ..."
```

需要补测试的场景：

- 排序、搜索、标签过滤、项目混合结果。
- 剪贴板兜底、自动粘贴失败、焦点恢复失败。
- 数据库 migration、损坏库隔离、备份恢复。
- 设置持久化、面板尺寸 / 位置 / 固定状态。
- 发布脚本参数、Sparkle 半配置、恢复脚本安全目标。

## 6. 文档同步规则

文档是交付物的一部分。以下改动必须同步文档：

| 改动 | 至少同步 |
| --- | --- |
| 功能、快捷键、UI 主链路 | README、[API 与功能说明](./API与功能说明.md)、[使用示例](./使用示例.md)、回归清单。 |
| 架构、依赖、系统集成 | [架构说明](./架构说明.md)、[技术选型](./技术选型.md)、[开发规范](./开发规范.md)。 |
| 数据库、配置、环境变量 | [配置说明](./配置说明.md)、[运维与排错指南](./运维与排错指南.md)。 |
| 打包、签名、公证、Sparkle | [部署说明](./部署说明.md)、[生产发布与恢复手册](./生产发布与恢复手册.md)。 |
| 产品定位、关键词、公开入口 | README、README.zh-CN、FAQ、llms.txt、llms-full.txt、[AI 搜索与可发现性](./ai-search-discoverability.md)、结构化元数据。 |
| 贡献流程或路线图 | CONTRIBUTING、[路线图与贡献指南](./路线图与贡献指南.md)。 |

提交前运行：

```bash
./scripts/check-docs.sh
```

## 7. 依赖规则

当前第三方依赖：

- GRDB.swift
- KeyboardShortcuts
- Sparkle

新增依赖前必须说明：

- 它解决的真实问题。
- 为什么现有依赖或系统 API 不够。
- 对启动速度、包体、权限、签名、公证、维护风险的影响。
- 是否会破坏“原生、本地、轻量”的产品边界。

## 8. PR 规则

- 一个 PR 只解决一个主题。
- 行为改动要有测试或明确的手动验证。
- UI 改动要说明对应的 `frontend-draft/` 文件。
- 文档改动要通过 `scripts/check-docs.sh`。
- 发布相关改动要跑 `release-readiness.sh`，并说明是否使用了 `--allow-build-only-tests` 降级。
- 不接受云同步、团队协作、远端 workflow orchestration 方向的 PR。
