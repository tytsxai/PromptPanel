## 当前发布基线

- 发布前先在“运行健康”确认：数据库路径正常、最近备份存在、近 7 天执行统计可见。
- 当前版本已接入 Sparkle 基础代码，但只有配置真实 `SUFeedURL` 和 `SUPublicEDKey` 后才启用更新检查；未配置时仍采用“本地打包 + 手动替换安装包”。
- `./scripts/build-app.sh` 默认生成 ad-hoc 签名 `.app` 与 zip，便于本机验证；面向外部分发前，仍需补齐正式 Developer ID 签名/公证链路。
- `./scripts/release-readiness.sh` 已提供一键发布就绪检查：会校验脚本语法、`scripts/check-docs.sh`、`swift build` / `swift test`、release 打包、签名校验、隔离数据目录启动冒烟，以及基于启动备份的恢复演练。
- 如果当前机器缺少 `xctest`，只能显式使用 `--allow-build-only-tests` 做降级预检；这不是对外公开分发的最终门禁。
- 发布门禁会按本次 `.app` 的 `CFBundleShortVersionString` / `CFBundleVersion` 定位对应 zip 并校验签名，避免输出目录里残留旧 zip 时误验错包。
- `./scripts/notarize-app.sh` 已提供公证/贴票据脚本；当机器已具备 Developer ID 签名和 `notarytool` keychain profile 后，可直接把已签名产物送审并完成 staple / Gatekeeper 复核。
- 每次启动会自动做一次启动维护：WAL checkpoint、清理 30 天前执行日志、按需生成启动备份、仅保留最近 7 份自动启动备份。
- 手动点击“立即备份”生成的备份不会被自动启动清理策略裁掉，可作为发版前保底备份。
- 如果数据库升级失败，应用现在会保留原始数据库并直接报错退出，不再把迁移错误误判成数据库损坏后自动重建空库。

## 发布产物生成

标准打包命令：

```bash
./scripts/build-app.sh
```

标准发布前自检命令：

```bash
./scripts/release-readiness.sh --output-dir /tmp/promptpanel-ready-check
```

如果当前机器只有 Command Line Tools、缺少 `xctest` runner，只能做本机预检：

```bash
./scripts/release-readiness.sh --allow-build-only-tests --output-dir /tmp/promptpanel-ready-check
```

这不是最终对外发布验收；正式发布前仍应在完整 Xcode 环境跑不带降级参数的检查。

常用参数：

```bash
./scripts/build-app.sh --sign-identity "Developer ID Application: Your Name (TEAMID)"
./scripts/build-app.sh --short-version 1.0.1 --build-version 2
./scripts/build-app.sh --sparkle-feed-url "https://example.com/appcast.xml" --sparkle-public-ed-key "BASE64_EDDSA_PUBLIC_KEY"
./scripts/build-app.sh --debug --no-archive
./scripts/release-readiness.sh --public-distribution --sign-identity "Developer ID Application: Your Name (TEAMID)"
./scripts/notarize-app.sh --app-path dist/PromptPanel.app --archive-path dist/PromptPanel-1.0.1+2-macos.zip --keychain-profile "AC_NOTARY"
```

产物位置：

- `.app`：`dist/PromptPanel.app`
- 压缩包：`dist/PromptPanel-版本号+构建号-macos.zip`

打包脚本当前会：

- 把 SwiftPM resource bundle 放入 `Contents/Resources`
- 把 `Sparkle.framework` 放入 `Contents/Frameworks`
- 为可执行文件补 `@executable_path/../Frameworks`
- 清理 `.app` 产物内的扩展属性，避免 `Sparkle.framework` 携带 `com.apple.FinderInfo` 等 xattr 导致严格签名校验失败
- 未传正式证书时做 ad-hoc 签名，便于本机 QA
- Sparkle feed URL 和 EdDSA 公钥必须成对注入，只提供其中一个时打包会失败
- 传入正式证书时执行 Developer ID 签名，但公证流程仍需单独跑通

## 标准发布流程

1. 在当前版本主界面点击“立即备份”，确认备份目录下生成新的 `.sqlite` 备份。
2. 先运行 `./scripts/release-readiness.sh`；如果只是本机 QA 且当前机器缺少 `xctest` runner，才允许加 `--allow-build-only-tests`；如果准备对外发包，至少加上 `--public-distribution --sign-identity ...` 做正式签名预检。
3. 生成已签名产物后，运行 `./scripts/notarize-app.sh --app-path ... --archive-path ... --keychain-profile ...`，完成 notarization、staple 和 Gatekeeper 复核。
4. 如需单独生成产物，运行 `./scripts/build-app.sh`，确认 `dist/PromptPanel.app` 和 zip 产物生成成功。
5. 退出应用。
6. 替换新的应用包。
7. 启动新版本。
8. 检查“运行健康”和“最近执行记录”。
9. 至少做一次真实执行，确认自动粘贴或复制兜底都正常。

补充说明：

- `--public-distribution` 当前只做“正式发包前硬门槛预检”，不会代替 Apple notarization / staple；这一步仍需在真实证书和账号环境下单独完成。
- 如果本机缺少 `xctest` runner，脚本会明确提示；只有显式传入 `--allow-build-only-tests` 才会降级为测试包构建校验，这种机器不应作为最终对外发布前的最后验收机。
- 发布冒烟阶段会在隔离目录创建数据库和启动备份，再用该备份恢复到隔离目标目录；这一步失败时，应优先排查备份结构、恢复脚本和 schema 兼容性，不要继续发版。

## 回滚流程

1. 退出应用。
2. 使用 `./scripts/restore-backup.sh /absolute/path/to/backup.sqlite` 恢复指定备份。
3. 重新启动应用并核对数据。

补充说明：

- `restore-backup.sh` 会先验证备份完整性和必要表结构，再把当前 `promptpanel.db` / `-wal` / `-shm` 移到 `Recovery/manual-restore-时间戳/`，最后写入目标备份。
- 如果你需要保留多个发版前人工备份，优先使用主界面“立即备份”，不要依赖自动启动备份替代。

## 数据异常恢复

- 如果启动时数据库损坏，应用会把原始文件移动到 `Recovery/recovered-时间戳/` 后自动重建新库。
- 如果启动时提示“数据库升级失败”，说明是迁移/代码问题，不是应用替你恢复成功了；此时应先停发当前版本并保留现场，不要强行重启覆盖。
- 数据库损坏自动恢复后，应优先从最近备份恢复，不要直接覆盖掉 `Recovery/` 中的现场文件。
- `Recovery/` 目录用于排障和人工比对，确认无用后再清理。
