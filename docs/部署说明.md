# PromptPanel 部署说明

更新时间：`2026-05-07`

## 1. 结论先说清楚

PromptPanel 的正式运行态只有一种：

- **本地 macOS 桌面应用**

用户要求里的“本地 / 容器 / 服务器”三种部署方式，需要按真实能力区分，而不是一概写成“支持”。

## 2. 部署方式矩阵

| 方式 | 当前状态 | 适用场景 | 不能做什么 |
| --- | --- | --- | --- |
| 本地 macOS 运行 | 推荐，且是唯一正式运行态 | 开发、日常使用、桌面验收、真实发布包运行 | 无 |
| 本地 macOS 打包/发布 | 推荐 | 构建 `.app`、签名、公证、回归、自测 | 不能替代最终用户安装验证 |
| macOS CI / 自托管 runner | 支持 | `swift build`、`swift test`、打包、签名校验、自动化预检 | 不能替代真实桌面人工兼容性验收 |
| Linux 容器运行产品 | 不支持 | 无 | 无法提供 `AppKit`、`NSPanel`、全局热键、登录启动、GUI 会话 |
| 普通服务器运行产品 | 不支持 | 无 | 不能成为终端用户的实际运行环境 |
| 服务器托管发布产物 | 支持 | 托管 `.zip`、`appcast.xml`、Sparkle 更新文件 | 不能运行 PromptPanel 本体 |

## 3. 为什么容器和服务器不是产品运行态

这是由项目能力边界决定的：

- 面板依赖 `NSPanel`
- 全局快捷键依赖 `KeyboardShortcuts` / macOS 桌面事件
- 自动粘贴依赖辅助功能权限和 GUI Session
- 登录启动依赖 `SMAppService`
- 用户价值来自“当前桌面前台应用上下文中秒开并注入内容”

所以：

- **容器** 最多只能做辅助构建或静态文件托管，不能替代客户端运行
- **服务器** 最多承担发布产物分发、Sparkle appcast 托管或 macOS CI 角色，不能代替用户本地客户端

## 4. 本地部署

### 4.1 环境要求

- macOS `14.0+`
- `zsh`
- Swift `5.10`
- 开发建议安装完整 Xcode
- 仅安装 Command Line Tools 时可以构建，但通常缺少 `xctest`；不能把 build-only 测试当成完整测试执行
- 如果要做对外发布，还需要：
  - Developer ID Application 证书
  - `notarytool` 可用环境
  - Apple notarization keychain profile

### 4.2 仓库结构要点

- 仓库根目录：脚本、CI、PRD、版本化 `docs/`
- SwiftPM 包根目录：仓库根目录

因此构建命令不要写错位置：

```bash
swift build
swift test
```

### 4.3 本地开发 / 构建

最基础构建：

```bash
swift build
```

测试：

```bash
swift test
```

如果本机缺少 `xctest`，只能做测试包构建校验：

```bash
swift build --build-tests
```

生成本地可打开的 `.app`：

```bash
./scripts/build-app.sh --debug --no-archive
open dist/PromptPanel.app
```

说明：

- `build-app.sh` 会把 SwiftPM 二进制和资源组装成真正的 `.app`
- 这是桌面验收时应优先使用的形态

### 4.4 本地发布前检查

标准入口：

```bash
./scripts/release-readiness.sh --output-dir /tmp/promptpanel-ready-check
```

它会统一执行：

1. shell 脚本语法检查
2. `scripts/check-docs.sh` 文档同步门禁
3. `swift build`
4. `swift test`
5. release `.app` + zip 打包
6. 签名校验
7. 隔离数据目录下的启动冒烟
8. 基于启动备份做隔离恢复演练

如果当前机器只有 Command Line Tools、缺少 `xctest`，可以显式降级为本机预检：

```bash
./scripts/release-readiness.sh --allow-build-only-tests --output-dir /tmp/promptpanel-ready-check
```

这只证明测试包可构建，不能作为对外发布前最终门禁。

如果只改文档、脚本说明、配置说明或 README，也要单独跑：

```bash
./scripts/check-docs.sh
```

这条门禁会检查必备文档、Markdown 本地链接、陈旧数据库/热键/部署说法、`PROMPTPANEL_*` 环境变量覆盖、顶层维护脚本是否被文档提到，以及 CI 是否包含文档同步检查。

如果准备做对外发包预检：

```bash
./scripts/release-readiness.sh \
  --public-distribution \
  --sign-identity "Developer ID Application: Your Name (TEAMID)"
```

### 4.5 本地正式打包 / 公证

正式打包：

```bash
./scripts/build-app.sh \
  --sign-identity "Developer ID Application: Your Name (TEAMID)"
```

补 Sparkle 更新元数据：

```bash
./scripts/build-app.sh \
  --sign-identity "Developer ID Application: Your Name (TEAMID)" \
  --sparkle-feed-url "https://example.com/appcast.xml" \
  --sparkle-public-ed-key "BASE64_EDDSA_PUBLIC_KEY"
```

公证与贴票据：

```bash
./scripts/notarize-app.sh \
  --app-path dist/PromptPanel.app \
  --archive-path dist/PromptPanel-1.0.0+1-macos.zip \
  --keychain-profile "AC_NOTARY"
```

### 4.6 本机安装验证

如果只是把当前本地版本安装到这台 Mac，推荐路径是：

```bash
./scripts/build-app.sh --output-dir dist/install-local
osascript -e 'quit app "PromptPanel"' || true
rm -rf /Applications/PromptPanel.app
cp -R dist/install-local/PromptPanel.app /Applications/PromptPanel.app
xattr -dr com.apple.quarantine /Applications/PromptPanel.app 2>/dev/null || true
open /Applications/PromptPanel.app
```

安装后要确认实际运行的是 `/Applications/PromptPanel.app`，不要误把 `dist/` 里的开发包当成已安装版本。

## 5. 容器部署

### 5.1 当前结论

- **不支持将 PromptPanel 本体部署到容器里运行**

### 5.2 原因

- Linux 容器没有可用的 AppKit 图形环境
- `NSPanel`、菜单栏、全局快捷键、登录启动都不是容器语义
- 自动粘贴需要用户本机辅助功能权限和图形焦点链路

### 5.3 容器在这个项目里还能干什么

只能做辅助能力，例如：

- 托管 Sparkle 的 `appcast.xml`
- 托管 `.zip` 发布包
- 托管说明页或内部下载页

如果团队一定要写“容器部署方案”，应明确写成：

- **容器用于分发，不用于运行产品**

## 6. 服务器部署

### 6.1 当前结论

- **不支持把服务器当 PromptPanel 的运行环境**

### 6.2 可以支持的服务器角色

服务器可以承担下面几类职责：

| 角色 | 是否适合 | 说明 |
| --- | --- | --- |
| 静态文件服务器 | 适合 | 托管 zip、`appcast.xml`、发布说明 |
| macOS CI 机器 | 适合 | 跑构建、测试、打包、公证 |
| 普通 Linux 业务服务器 | 部分适合 | 只能做静态托管或辅助发布，不能运行产品 |
| 远程桌面常驻“替用户运行” | 不适合 | 不符合产品设计，也不稳定 |

### 6.3 推荐的服务器化落地方式

如果需要“服务器参与部署”，推荐这样设计：

```mermaid
flowchart LR
    DEV["开发机 / macOS Runner"] --> BUILD["构建 + 签名 + 公证"]
    BUILD --> ART["静态产物托管: zip + appcast.xml"]
    ART --> CLIENT["用户本地 macOS PromptPanel.app"]
```

这才是和当前架构一致的“服务器参与方式”。

## 7. 推荐部署拓扑

### 开发态

- 一台本地 macOS 开发机
- 本地运行 `.app`
- 随代码维护的 `docs/`
- GitHub Actions 做基础回归

### 对外分发态

- 一台 macOS 发布机或 macOS runner
- 一个静态文件托管位置：
  - Nginx
  - S3 + CDN
  - GitHub Releases + 自建 appcast

### 不推荐

- 试图用 Docker 代替客户端运行
- 试图让后端服务器“远程帮用户粘贴”
- 没有完整 Xcode / 证书 / notarization 条件却直接发公网上线包

## 8. 部署时的最低验收标准

无论你采用哪种交付方式，最终都要满足下面几点：

- 打包产物是完整 `.app`，不是裸二进制
- `codesign --verify --deep --strict` 通过
- zip 解压后仍能通过签名校验
- `scripts/check-docs.sh` 通过，文档没有落后于脚本/配置/部署边界
- 应用首次启动能创建数据库和启动备份
- 主界面的“运行健康”可用
- 至少完成一轮真实桌面目标应用兼容性验收

如果这些没过，就不能把“服务器已部署”“容器已准备好”当成交付完成。
