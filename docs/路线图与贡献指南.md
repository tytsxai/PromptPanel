# PromptPanel 路线图与贡献指南

更新时间：`2026-05-07`

本文说明 PromptPanel 的长期发展方向、明确非目标、贡献入口和维护节奏。具体 PR 规则见 [CONTRIBUTING](../CONTRIBUTING.md)。

## 1. 产品路线原则

PromptPanel 的路线不是把功能做满，而是长期守住一个高频工作流：

```text
全局快捷键 -> 搜索 -> 回车 -> 内容可靠进入当前输入框
```

路线判断标准：

- 是否明显提升这条主链路的速度、可靠性或可维护性。
- 是否保持本地优先和单用户模型。
- 是否能在 macOS 原生能力里简单落地。
- 是否会增加数据泄漏、权限扩大、发布复杂度或长期维护负担。

## 2. 当前已完成基线

v0.1 基线：

- 全局快捷键唤出快捷面板。
- 当前项目 + `通用项目` 混合搜索。
- SQLite + FTS5 本地存储与搜索。
- 词条类型、标签、置顶、排序、使用次数等级。
- 剪贴板优先、自动粘贴尽力而为、失败可诊断。
- 主窗口内容库、项目管理、设置、运行健康。
- 开机启动、主题、面板固定、面板尺寸与位置持久化。
- Sparkle 更新集成。
- build、release readiness、notarization、restore 脚本。
- README、docs、FAQ、贡献指南、issue/PR 模板、llms 索引和结构化元数据。

## 3. 近期路线

| 优先级 | 方向 | 接受条件 |
| --- | --- | --- |
| P1 | JSON / Markdown 导入导出 | 本地文件，不依赖云；格式可审计；有测试和恢复说明。 |
| P1 | 重复执行上一条词条 | 不影响现有搜索路径；避免误触；需要可见状态或清晰快捷键。 |
| P1 | 搜索和标签体验细化 | 不牺牲延迟；更新 FTS / tag 契约、测试和文档。 |
| P2 | 更多目标 app 兼容性样本 | 记录真实 app、版本、结果和失败原因，不用猜测替代证据。 |
| P2 | 文档站或 GitHub Pages | 必须复用 README/docs/llms 元数据，不能另起一套容易漂移的内容。 |
| P3 | 变量模板 | 只有在交互足够轻、不拖慢主链路、不引入工作流编排时才考虑。 |

## 4. 永久非目标

不会接受的方向：

- 云同步。
- 团队协作。
- 多用户权限。
- 远程执行。
- 工作流编排。
- 浏览器插件替代原生 app。
- 把 PromptPanel 变成知识库服务器或 agent 平台。

这些方向即使实现质量高，也会破坏项目定位。类似需求应做成独立项目，而不是合进 PromptPanel。

## 5. Issue 贡献

提交 bug 时请提供：

- PromptPanel 版本或 commit。
- macOS 版本和 CPU 架构。
- 目标 app，例如 ChatGPT web、Claude web、Cursor、VS Code、Terminal。
- 粘贴结果：已复制并粘贴、仅复制、没有反应、粘到错误位置。
- 最近执行日志，必须移除私密 Prompt 内容。

提交功能建议时请说明：

- 它解决的重复工作流或失败场景。
- 为什么现有功能无法覆盖。
- 是否保持本地优先。
- 是否会触碰云同步、团队协作或工作流编排边界。
- 你愿意接受的最小可用版本。

## 6. PR 贡献

推荐 PR 类型：

- 小型 bug 修复。
- 搜索、排序、执行、恢复等核心路径的测试补强。
- 目标 app 兼容性样本补充。
- 文档、示例、FAQ、AI 搜索索引维护。
- 不扩大产品边界的 UI 可用性优化。
- 打包、签名、公证、恢复脚本的安全性改进。

PR 必须说明：

- 改了什么。
- 为什么符合当前产品边界。
- 运行了哪些验证命令。
- 是否更新了 docs/README/FAQ/llms/metadata。
- 如果是 UI，先改了哪个 `frontend-draft/` 文件。
- 如果使用 `--allow-build-only-tests`，明确说明这是降级验证，不是完整测试。

## 7. 验证矩阵

| 改动类型 | 最低验证 |
| --- | --- |
| 文档 / SEO / metadata | `./scripts/check-docs.sh` |
| Swift 编译相关 | `swift build` |
| 行为改动 | `swift test` |
| UI 改动 | `swift build` + 打包 `.app` smoke + 截图或说明 |
| 数据库 / 恢复 | 相关测试 + 恢复 dry run + 文档同步 |
| 发布脚本 | `zsh -n` + `./scripts/release-readiness.sh --output-dir /tmp/promptpanel-ready-check` |
| 对外分发 | 不降级 release readiness + Developer ID 签名 + notarization + Gatekeeper 复核 |

## 8. 长期维护节奏

建议保持以下节奏：

- 每个功能 PR 都带文档同步。
- 每次 release 前更新 `CHANGELOG.md`。
- 每次定位变化都同步 `README.md`、`README.zh-CN.md`、`FAQ.md`、`llms.txt`、`llms-full.txt`、`codemeta.json`、`docs/search-metadata.schema.jsonld`。
- 每次搜索、执行、权限、数据库和发布链路变更都补回归记录。
- 每季度重新检查 GitHub topics、README 首屏、FAQ 和 AI 搜索描述是否仍符合项目真实状态。

## 9. 贡献者上手路径

1. 阅读 [项目介绍](./项目介绍.md) 和 [README](../README.md)。
2. 阅读 [架构说明](./架构说明.md) 和 [API 与功能说明](./API与功能说明.md)。
3. 本地运行 `swift build` 和 `./scripts/check-docs.sh`。
4. 从 bug、测试、文档或兼容性样本开始。
5. 非小型功能先开 issue，不要直接发大 PR。
