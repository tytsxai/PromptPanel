# 项目快贴 V1 技术选型

调研日期：`2026-04-18`

说明：本文中的 API、框架能力、发布方式等信息来自官方文档或官方仓库；“最终推荐哪套方案”属于结合当前 PRD 做出的工程判断，不是某个官方文档直接给出的结论。

## 1. 背景与结论

基于当前 PRD，这个产品不是普通桌面 CRUD 应用，而是一个**强依赖 macOS 原生系统能力**的本地效率工具。它的核心主链路成败主要取决于以下几点：

- 全局快捷键是否稳定
- 面板是否能以足够低延迟呼出并抢到焦点
- 剪贴板写入与自动粘贴是否稳定可控
- 权限引导是否清晰
- 常驻、唤起、关闭、回到原工作流是否顺滑

结合项目现状与联网调研，**V1 最优技术选型建议是：**

- 客户端框架：`Swift + AppKit 为主，SwiftUI 为辅`
- 本地数据库：`SQLite`
- Swift 数据访问层：`GRDB.swift`
- 全局快捷键：`KeyboardShortcuts` 或直接封装 Carbon Hot Key
- 菜单栏与常驻：`NSStatusItem`
- 快捷面板：`NSPanel`
- 剪贴板：`NSPasteboard`
- 自动粘贴：`CGEvent` 模拟粘贴按键，失败时明确走剪贴板兜底
- 开机启动：`SMAppService`
- 自动更新：`Sparkle 2`
- 发布方式：`Developer ID + Notarization`，优先站外分发，不走 Mac App Store

一句话结论：

**V1 应该做成原生 macOS 应用，不建议用 Tauri，也不建议用 Electron 作为主实现。**

原因不是“原生更优雅”，而是这个产品的关键能力本来就落在 macOS 系统集成层，跨平台壳不会带来核心收益，反而会增加权限、焦点控制、窗口行为和排障复杂度。

---

## 2. 选型判断标准

这次选型不按“团队熟不熟 Web”来判断，主要按是否能把主链路做稳来判断。核心评估维度如下：

### 2.1 系统能力适配度

必须直接覆盖以下能力：

- 全局快捷键
- 常驻菜单栏
- 浮层/面板窗口
- 剪贴板访问
- 跨应用自动粘贴
- 无障碍权限检测与引导
- 开机启动
- 站外安装与更新

### 2.2 主链路时延

PRD 明确要求：

- 面板从快捷键触发到可输入搜索，目标不超过 `300ms`
- 搜索结果刷新目标不超过 `100ms`
- 词条执行从确认到完成复制，目标不超过 `200ms`

因此框架必须在以下路径上足够轻：

- 冷启动
- 常驻内存占用
- 窗口显示与聚焦
- 列表搜索与排序

### 2.3 稳定性与可调试性

这个产品一旦出现下面的问题，用户会直接判定不可用：

- 热键失灵
- 面板弹了但焦点不对
- 执行后没有成功复制
- 自动粘贴在部分应用中行为混乱
- 权限没给时缺少明确提示

所以选型要优先考虑：

- 出问题时能否快速定位
- 是否容易按 macOS 真实行为修复
- 是否能做降级与兜底

### 2.4 后续扩展性

V1 不做云同步和协作，但后续大概率会做：

- 模板变量
- 导入导出
- 项目自动识别
- 快捷面板更复杂的排序逻辑
- 使用统计与推荐

因此数据层和 UI 架构要留出空间，但不能为了未来把 V1 复杂化。

## 3. 联网调研摘要

下面只保留对这次决策真正有价值的信息。

### 3.1 Apple 官方能力与约束

Apple 官方文档覆盖了本产品所需的大部分关键原生能力：

- `NSPanel`：适合实现浮层式快捷面板
- `NSStatusItem`：适合实现菜单栏常驻入口
- `NSPasteboard`：用于稳定写入剪贴板
- `SMAppService`：用于登录启动项管理
- `NSHostingController` / SwiftUI 与 AppKit 混用能力：适合“窗口与系统行为用 AppKit，业务界面用 SwiftUI”
- `CGEvent`：可用于构造和发送键盘事件
- `AXIsProcessTrustedWithOptions`：用于检测并触发无障碍授权引导
- Apple Notarization 文档：明确支持站外分发的 macOS 应用做公证发布

这些能力说明了一点：**这类产品在 Apple 官方能力体系里是有标准路径的。**

### 3.2 Tauri 官方能力现状

Tauri v2 官方文档和插件体系已经覆盖不少桌面能力，包括：

- Global Shortcut
- Tray / Tray Icon
- SQL Plugin
- Clipboard 相关插件
- 权限能力模型（capabilities）
- 基于系统 WebView 的桌面渲染

Tauri 的优点很明确：

- 比 Electron 更轻
- Rust 后端能力强
- 可以用前端栈快速做管理界面

但对这个项目来说，问题不在“有没有插件”，而在于以下现实：

- 主链路问题最终仍要落回 macOS 原生行为排障
- 浮层窗口、焦点恢复、跨应用粘贴、权限引导这些问题在跨层封装下更难调
- V1 明确只做 macOS，本身不需要跨平台红利
- 一旦插件行为与实际需求有偏差，还是要自己补原生桥接

结论是：**Tauri 可以做，但不是这个产品的最优解。**

### 3.3 Electron 官方能力现状

Electron 官方文档明确提供：

- `globalShortcut`
- `Tray`
- `clipboard`
- 多进程模型
- 性能优化指导

Electron 当然也能做出这个产品，但它的问题同样明确：

- 运行时更重
- 常驻应用内存占用更高
- 冷启动与窗口唤起成本通常更大
- 你仍然要处理 macOS 系统行为，只是外面再套了一层 Chromium/Node

对于一个“菜单栏常驻 + 快捷面板 + 高频秒开”的工具，Electron 的工程体验优势，不足以抵消其运行成本和系统集成复杂度。

结论是：**Electron 不适合作为这个项目 V1 的主选型。**

### 3.4 本地搜索与存储

SQLite 官方文档中的 `FTS5` 很适合这个产品：

- 本地全文搜索
- 标题和内容搜索
- 排序时可叠加业务分值
- 数据量在个人本地场景下完全足够

`WAL` 模式也适合桌面本地应用：

- 读写并发更友好
- 日常响应更稳

在 Swift 侧，`GRDB.swift` 是成熟的 SQLite 封装，适合做：

- schema migration
- typed query
- FTS 表管理
- 本地事务一致性

## 4. 方案对比

## 4.1 候选方案

本次主要比较三类方案：

1. `Swift + AppKit/SwiftUI`
2. `Tauri v2 + Rust + Web UI`
3. `Electron + Web UI`

## 4.2 结论矩阵

| 维度 | Swift 原生 | Tauri | Electron |
| --- | --- | --- | --- |
| macOS 系统能力适配 | 最强 | 中等偏强 | 中等 |
| 全局快捷键稳定性 | 最强 | 依赖插件与桥接 | 可用 |
| 面板窗口/焦点控制 | 最强 | 中等 | 中等 |
| 剪贴板与自动粘贴排障 | 最直接 | 需要跨层定位 | 需要跨层定位 |
| 常驻资源占用 | 最低 | 较低 | 最高 |
| 管理界面开发速度 | 中等 | 高 | 高 |
| V1 落地风险 | 最低 | 中等 | 中高 |
| 后续跨平台可能性 | 低 | 高 | 高 |
| 对当前 PRD 的匹配度 | 最高 | 中等 | 较低 |

## 4.3 为什么推荐 Swift 原生

### 原因一：主问题是系统集成，不是页面开发

这个产品真正难的地方不是做一个列表页面，而是：

- 热键注册
- 面板弹出与聚焦
- 权限检测与引导
- 自动粘贴兼容性
- 失败兜底与回流原工作流

这些都属于 macOS 原生问题。既然问题核心在原生层，V1 就不应该再额外引入跨平台壳。

### 原因二：性能目标更容易稳定达成

PRD 对唤起速度要求比较明确。原生常驻应用更容易做到：

- 常驻内存小
- 面板秒开
- 搜索与选择路径延迟更低

### 原因三：问题定位最短路径

当用户反馈“某个应用里自动粘贴失败”时，最有效的定位路径是直接看：

- 权限状态
- 前台应用
- 焦点切换时序
- `CGEvent` 是否成功发送
- 面板是否过晚关闭

如果外面套了一层 Tauri/Electron，排障链路会更长，根因修复也更慢。

### 原因四：V1 没有跨平台收益

当前 PRD 已经明确是 `Mac 个人本地版`。既然目标不是 Windows 也不是 Web，就没有必要为不存在的跨平台需求承担额外架构复杂度。

## 5. 推荐技术方案

## 5.1 总体架构

推荐采用以下结构：

- UI 层：`AppKit + SwiftUI`
- 应用外壳与系统集成：`AppKit`
- 业务界面：`SwiftUI`
- 数据层：`SQLite + GRDB.swift`
- 搜索层：`SQLite FTS5 + 业务排序`
- 系统能力层：`Hotkey / Clipboard / Accessibility / Paste / Tray / LoginItem`

建议的职责分层：

1. `AppShell`
   负责应用生命周期、菜单栏、主窗口、快捷面板、权限引导。

2. `HotkeyService`
   负责注册、更新、监听全局快捷键。

3. `PanelService`
   负责快捷面板的显示、隐藏、焦点恢复、Esc 关闭、点击外部关闭。

4. `EntrySearchService`
   负责词条检索、排序、项目与通用项目混排。

5. `ExecuteService`
   负责“写剪贴板 -> 尝试粘贴 -> 提示结果 -> 关闭面板”。

6. `PermissionService`
   负责检测无障碍权限、引导用户开启、输出当前可用能力。

7. `ProjectRepository / EntryRepository`
   负责项目与词条增删改查、迁移、排序和统计。

## 5.2 为什么是 AppKit 为主、SwiftUI 为辅

建议不要全 SwiftUI 硬做。

原因如下：

- 菜单栏、面板、焦点控制、窗口层级这些地方，AppKit 更直接
- 主界面的表单、列表、设置页，用 SwiftUI 更省开发成本
- 混合方案能兼顾系统控制力和开发效率

实际落地建议：

- `NSApplication` / `NSStatusItem` / `NSPanel` / window behavior 用 AppKit
- 主界面内容和词条管理页用 SwiftUI
- 通过 `NSHostingController` 承接 SwiftUI 页面

这比“全 AppKit”更高效，也比“全 SwiftUI”更稳。

## 5.3 数据存储选型

选择：

- `SQLite` 单文件数据库
- `GRDB.swift`
- `FTS5` 作为搜索索引

理由：

- 本地单机场景最合适
- 无需引入服务端或额外进程
- 搜索性能足够
- schema migration 清晰
- 后续做导入导出、备份恢复都方便

建议的核心表：

- `projects`
- `entries`
- `entry_search` 或 `entries_fts`
- `app_settings`

`entries` 至少应包含：

- `id`
- `project_id`
- `title`
- `content`
- `type`
- `is_pinned`
- `sort_order`
- `use_count`
- `last_used_at`
- `created_at`
- `updated_at`

搜索排序建议不要只依赖 FTS 分数，实际应采用混合排序：

- 置顶优先
- 当前项目优先于通用项目
- 搜索匹配分
- 最近使用时间
- 使用次数
- 手动排序值

## 5.4 全局快捷键

优先建议：

- V1 使用 `KeyboardShortcuts`

原因：

- 社区成熟
- Swift 集成简单
- 适合快速完成设置与持久化

备选：

- 直接封装 Carbon `RegisterEventHotKey`

什么时候考虑自己封装：

- 需要更细粒度控制
- 依赖包行为不满足需求
- 后续想减少第三方依赖

V1 没必要一上来自己啃全部底层细节，先把主链路做通更重要。

## 5.5 自动粘贴策略

推荐执行策略如下：

1. 词条内容写入 `NSPasteboard`
2. 关闭面板或至少让面板退出前台焦点
3. 检查无障碍权限状态
4. 若权限可用，则发送 `Cmd+V`
5. 若发送失败或处于不兼容应用，则提示“已复制，可手动粘贴”

这里要明确区分两件事：

- **根因方案**：自动粘贴依赖系统权限与目标应用兼容性，本质上不是 100% 可控能力
- **产品兜底**：必须始终保证已经成功复制到剪贴板

建议不要在 V1 做这些高风险动作：

- 自动回车发送
- 自动恢复用户原剪贴板
- 复杂注入方式的多应用特判

这些都会显著提高副作用和兼容性风险。

## 5.6 搜索实现

V1 搜索建议采用：

- 标题和内容统一进入 FTS5
- 搜索输入防抖可以非常轻，甚至不做显式防抖
- 每次输入实时查询
- 结果数控制在一个合理窗口内，例如前 `50` 条

在个人本地数据量下，这个方案足够快，也足够简单。

## 5.7 发布与更新

推荐：

- 站外分发 `.dmg`
- `Developer ID` 签名
- `Notarization`
- 使用 `Sparkle 2` 做应用内更新

不建议 V1 走 Mac App Store，原因：

- 本产品涉及无障碍、全局快捷键、跨应用控制
- 这类能力在商店审核与沙盒限制下更容易受限
- 站外分发对这类工具更现实

## 6. 不推荐方案说明

## 6.1 为什么不推荐 Tauri 作为 V1 主实现

不是说 Tauri 不行，而是它对这个产品没有形成决定性优势。

主要问题：

- 你仍然绕不开 macOS 原生层问题
- 快捷面板、焦点、跨应用粘贴这类问题跨越 WebView、Rust、插件、系统层，定位更长
- V1 只有 macOS，没有跨平台收益
- 管理界面虽然会更快，但本项目最难的不是管理界面

适合 Tauri 的前提是：

- 你明确要做 Windows/macOS 双端
- 或者这个产品主要是管理后台式桌面应用，系统集成只是边缘能力

当前都不是。

## 6.2 为什么不推荐 Electron

Electron 的问题更直接：

- 资源占用偏高
- 常驻类工具的体验成本更明显
- 对“秒开面板”的目标不占优
- 同样不能减少你对 macOS 真实行为的理解和处理

如果未来做团队版、强富文本、复杂可视化管理后台，Electron 才可能重新进入候选。

## 7. 建议的工程实现方案

## 7.1 技术栈清单

建议最终落地为：

- 语言：`Swift 5.10+`
- IDE：`Xcode 16+`
- 最低系统版本：建议 `macOS 14+`
- UI：`AppKit + SwiftUI`
- 数据库：`SQLite`
- ORM / Query：`GRDB.swift`
- 热键：`KeyboardShortcuts`
- 更新：`Sparkle`
- 包管理：`Swift Package Manager`
- 测试：`XCTest`

### 为什么建议 `macOS 14+`

如果目标用户是重度 AI 用户与开发者，系统版本通常不会太旧。把最低版本提到较新的 macOS，有几个现实好处：

- 权限与系统行为差异更少
- SwiftUI/AppKit 混用更稳
- 发布与维护成本更低

如果后续发现用户确实需要更低版本，再评估是否降到 `macOS 13`。

## 7.2 建议目录结构

```text
PromptPanel/
  App/
    PromptPanelApp.swift
    AppDelegate.swift
  Core/
    Models/
    Repositories/
    Services/
    Database/
    Utils/
  Features/
    Panel/
    MainWindow/
    Projects/
    Entries/
    Settings/
    Permissions/
  Integrations/
    Hotkey/
    Tray/
    Clipboard/
    Paste/
    LoginItem/
    Updater/
  Resources/
  Tests/
```

这个结构的重点不是“好看”，而是把系统集成层与业务层拆开，后面排障不会乱。

## 7.3 核心状态设计

建议有一个统一的 `AppState`，至少包含：

- 当前项目 ID
- 通用项目 ID
- 面板显示状态
- 主窗口显示状态
- 权限状态
- 热键配置

但不要把所有业务都塞成一个巨大的全局单例。仓储、服务、窗口控制仍然应拆分。

## 8. 首版实施建议

## 8.1 第一阶段：先打通主链路

先只做以下能力：

- 菜单栏常驻
- 全局快捷键呼出/关闭面板
- 面板搜索和键盘选择
- SQLite 存储
- 项目与词条基础 CRUD
- 写剪贴板
- 有权限时尝试自动粘贴
- 无权限或失败时明确提示

这阶段不要急着做：

- 漂亮动画
- 复杂设置项
- 导入导出
- 变量模板
- 剪贴板恢复

## 8.2 第二阶段：补齐可靠性

在主链路通后，优先补这些：

- 权限引导页
- 兼容性提示
- 执行日志与失败原因记录
- 项目删除前校验与迁移流程
- 排序与搜索体验优化

## 8.3 第三阶段：补发布能力

包括：

- Developer ID 签名
- Notarization
- dmg 打包
- Sparkle 自动更新
- 崩溃与基础日志收集

## 9. 风险与应对

## 9.1 最大风险不是数据库和界面，而是自动粘贴兼容性

这件事必须提前接受现实：

- 自动粘贴在不同应用中兼容性不同
- 某些应用里可能只能保证复制成功

因此文案、提示和降级路径必须设计清楚。不要把产品承诺写成“任何应用都稳定自动输入”。

## 9.2 权限是核心门槛

没有无障碍权限时，自动粘贴能力会受限。V1 必须做到：

- 首次启动明确说明用途
- 可随时查看当前权限状态
- 权限缺失时能继续走复制兜底

## 9.3 不能忽略日志能力

如果后续线上遇到“某些应用粘贴失败”，没有日志基本无法快速定位。

建议至少记录：

- 执行时间
- 前台应用 Bundle ID
- 是否有无障碍权限
- 剪贴板写入是否成功
- 自动粘贴是否尝试
- 最终是否走兜底

注意日志里不要直接明文持久化用户词条内容，避免隐私风险。可以只记录词条 ID 和执行结果。

---

## 10. 最终建议

最终推荐技术方案如下：

- **框架**：`Swift + AppKit 为主，SwiftUI 为辅`
- **数据库**：`SQLite + GRDB.swift`
- **搜索**：`SQLite FTS5`
- **热键**：`KeyboardShortcuts`
- **常驻入口**：`NSStatusItem`
- **快捷面板**：`NSPanel`
- **自动粘贴**：`NSPasteboard + CGEvent`
- **开机启动**：`SMAppService`
- **更新**：`Sparkle`
- **发布**：`Developer ID + Notarization + DMG`

不推荐：

- `Tauri` 作为 V1 主实现
- `Electron` 作为 V1 主实现

核心判断只有一句：

**这个产品最关键的是把 macOS 原生交互链路做稳，而不是尽快做出一个跨平台桌面壳。**
