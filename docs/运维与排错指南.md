# PromptPanel 运维与排错指南

更新时间：`2026-05-09`

## 1. 先判断问题属于哪一类

PromptPanel 常见问题基本分成五类：

1. 构建 / 测试环境问题
2. 面板 / 热键 / 焦点问题
3. 自动粘贴 / 权限 / 目标应用兼容性问题
4. 数据库 / 备份 / 恢复问题
5. 签名 / 打包 / 公证 / 更新问题

第一步不要直接改代码，先把问题归类。

## 2. 先看哪些证据

### 2.1 应用内证据

优先看主界面的：

- `运行健康`
- `最近执行记录`

这里能直接看到：

- 数据库路径
- 备份目录
- 最近备份
- 最近 7 天执行汇总
- 每条执行的目标应用、结果、失败原因、耗时

### 2.2 文件系统证据

默认目录：

- 数据：`~/Library/Application Support/PromptPanel`
- 日志：`~/Library/Logs/PromptPanel`
- 备份：`~/Library/Application Support/PromptPanel/Backups`
- 恢复现场：`~/Library/Application Support/PromptPanel/Recovery`

### 2.3 命令行证据

最先跑这三个：

```bash
./scripts/check-docs.sh
swift build
swift test
./scripts/release-readiness.sh --output-dir /tmp/promptpanel-ready-check
```

如果第三个命令都过不了，就不要先纠结 UI 现象。

## 3. 快速健康检查

每次接手、上线前或大改后，先过这一套：

1. `git status` 看工作区是否干净，尤其注意 `scripts/` 是否有未提交变更。
2. `./scripts/check-docs.sh`
3. `swift build`
4. `./scripts/release-readiness.sh --output-dir /tmp/promptpanel-ready-check`
5. 打开打包后的 `.app`，确认应用能稳定驻留。
6. 看主界面“运行健康”是否能读到数据库和备份。
7. 做一次真实词条执行，确认执行日志新增记录。

## 3.1 Computer Use 调试入口

如果你要用 `Computer Use` 直接抓 PromptPanel，不要默认双击 `.app` 后再连。

原因：

- PromptPanel 正常启动是 `LSUIElement` 形态
- 没有主动打开主窗口/面板时，`Computer Use` 很容易拿不到可见窗口，只会看到进程或直接超时

统一用这条命令启动：

```bash
./scripts/launch-computer-use.sh --surface library
```

常用变体：

- `./scripts/launch-computer-use.sh --surface settings`
- `./scripts/launch-computer-use.sh --surface panel`
- `./scripts/launch-computer-use.sh --surface library --skip-build`

这条脚本会自动：

- 用隔离的 `AppSupport` / `Logs` 目录启动，避免污染真实数据
- 给 QA 包注入独立的 Bundle ID / 显示名，避免和正式实例共用同一标识
- 以可见窗口形态打开主窗口或面板
- 输出本次 QA 根目录，便于后续查数据库和日志

脚本拉起后，`Computer Use` 里直接连脚本输出的目标标识；当前默认是 `com.promptpanel.app.computeruse`。

## 4. 常见问题排查表

| 现象 | 大概率原因 | 先检查什么 | 处理建议 |
| --- | --- | --- | --- |
| `swift build` 失败 | 依赖、SwiftPM 状态、本机 toolchain 问题 | `swift --version`、`Package.swift`、`.build` 状态 | 先恢复基础构建能力，再看功能问题 |
| `scripts/check-docs.sh` 失败 | 文档链接断裂、关键脚本/环境变量未被文档覆盖、出现旧数据库名/旧热键/错误部署说法 | 脚本输出的文件与命中行、最近改过的 README/docs/scripts/CI | 先按脚本输出修文档或同步矩阵，再继续发布门禁；不要只改脚本绕过文档问题 |
| `swift build` 卡在 GitHub 依赖拉取 | 本机 Git 全局 `insteadOf` 把 `https://github.com/` 改写成 SSH，或另一个 SwiftPM 进程占用 `.build` | `git config --global --get-regexp 'url\\..*insteadOf'`、`ps aux | rg '[s]wift|git-upload-pack'` | 等并发 SwiftPM 结束；必要时临时用 `GIT_CONFIG_GLOBAL=/dev/null swift build` 验证，避免把 SSH 权限问题误判为源码失败 |
| `swift test` 只构建不真正执行 | 机器只有 Command Line Tools，没有 `xctest` runner | `xcrun --find xctest`、`release-readiness.sh` 是否显式使用 `--allow-build-only-tests` | 安装完整 Xcode；不要把“测试包编译成功”当成真实测试已跑完 |
| 新启动实例马上退出 | 已有同 bundle id 的 PromptPanel 实例在运行 | `ps aux | rg PromptPanel`、当前包的 bundle id、是否设置 `PROMPTPANEL_ALLOW_EXISTING_INSTANCE` | 关闭旧实例或用隔离 bundle id；正式运行不要绕过单实例保护 |
| 发布检查卡在恢复演练 | 启动备份不可恢复、schema 校验失败、`restore-backup.sh` 与当前库结构漂移 | `release-readiness.sh` 输出、隔离 QA 目录、`sqlite3 PRAGMA integrity_check` | 先修备份/恢复链路，再发包；不要绕过恢复演练继续上线 |
| 热键没反应 | 应用未运行、热键冲突、快捷键被改掉 | 菜单栏图标、设置页快捷键展示、执行日志有无新样本 | 先确认应用在跑，再确认当前快捷键是不是 `⌥2` 或被用户改过 |
| 面板出来了但不能马上输入 | 焦点链路不稳、未以最终 `.app` 形态验收 | `.app` 启动形态、面板相关诊断日志 | 先用打包 `.app` 复现，不要拿终端直跑二进制替代最终桌面结论 |
| 执行后没自动粘贴 | 无辅助功能权限、目标应用未恢复焦点、`CGEvent` 失败 | 最近执行记录里的 `failure_reason`、`front_app_bundle_id`、`observed_app_bundle_id` | 按失败原因分流，不要统一叫“粘贴失败” |
| 只进入复制模式 | 权限未授权或执行链路主动降级 | 执行日志 `accessibility_not_granted` / `target_app_not_restored` | 先授权或补兼容性样本，再决定是否改代码 |
| Sparkle “检查更新”不可用 | feed URL 或公钥未配置 | 设置页更新状态、环境变量、打包注入参数 | 先把配置补齐，再看更新逻辑 |
| 启动提示数据库升级失败 | migration 问题，不是损坏库 | 应用报错、数据库现场、最近代码变更 | 停止继续启动覆盖现场，先保留数据库并检查 migration |
| 启动后提示已自动恢复数据库 | 库文件损坏 | `Recovery/recovered-*`、最近备份 | 优先从备份恢复，不要直接删恢复现场 |
| `restore-backup.sh` 拒绝恢复 | 备份损坏、不是 PromptPanel 库、staging 副本校验失败、应用仍在运行 | 脚本输出、`sqlite3` integrity check、进程状态、`Recovery/manual-restore-staging-*` 是否残留 | 先修备份来源，不要绕过校验强灌 |
| `codesign --verify` 失败 | 打包不完整、签名顺序不对、framework/resource 缺失、产物携带 `com.apple.FinderInfo` 等扩展属性 | `build-app.sh` 输出产物、Frameworks/Resources 内容、`xattr -lr dist/.../PromptPanel.app` | 先跑 `release-readiness.sh`，不要手工乱拷贝包内容 |
| 发布检查通过但分发包版本不对 | 输出目录残留旧 `PromptPanel-*.zip`、人工挑错包 | `release-readiness.sh` 输出、`.app` 的 `CFBundleShortVersionString` / `CFBundleVersion`、实际 zip 文件名 | 发布门禁会按本次 `.app` 的版本号精确验 zip；人工发包也必须使用同名新 zip |
| 打包时 Sparkle 参数报缺失 | 只传了 feed URL 或只传了 EdDSA 公钥 | `--sparkle-feed-url`、`--sparkle-public-ed-key` 参数或环境注入 | 两个参数必须成对来自同一套 appcast 配置；否则先补配置，不要绕过检查 |
| 公证失败 | 证书、keychain profile、archive 形态不对 | `notarytool` 输出、证书、archive 路径 | 先修发布环境，不要把它归因到业务代码 |

## 5. 自动粘贴问题的标准排查顺序

这是最容易被误判的一类问题，必须按顺序查：

1. 看执行日志有没有新记录。
2. 看结果是不是 `clipboard_only`。
3. 看 `failure_reason` 是什么。
4. 如果是 `accessibility_not_granted`：
   - 去系统设置确认权限
   - 重新执行一次
5. 如果是 `target_app_not_restored`：
   - 看目标应用是否真的回到前台
   - 看是否涉及切 Space、全屏窗口、输入法、特殊编辑器
6. 如果是 `paste_event_creation_failed`：
   - 优先查系统事件派发链路
7. 如果日志显示 `success` 但目标应用没落值：
   - 归入目标应用兼容性样本
   - 补到 [兼容性回归记录](./兼容性回归记录.md)

不要直接用“权限看起来开了”“我感觉焦点已经回去了”代替证据。

## 6. 数据库问题处理顺序

### 6.1 启动失败或报迁移错误

正确顺序：

1. 先停止继续重试启动
2. 备份当前 `promptpanel.db`、`-wal`、`-shm`
3. 查看是否是 migration 失败而不是损坏恢复
4. 检查最近代码是否改过 schema / migration
5. 只在明确知道后果时才使用破坏性调试开关

### 6.2 数据库损坏已被自动隔离

正确顺序：

1. 记录 `Recovery/recovered-*` 路径
2. 检查最近备份是否可用
3. 优先使用 `restore-backup.sh` 恢复
4. 比对恢复后运行健康和主界面数据
5. 确认无误后再处理恢复现场

### 6.3 手动恢复

标准命令：

```bash
./scripts/restore-backup.sh /absolute/path/to/backup.sqlite
```

脚本已经内置了最重要的保护：

- 坏备份拒绝
- 非 PromptPanel 结构拒绝
- 应用运行中拒绝
- 先复制到 staging 并二次 integrity check，确认恢复源可用后才转移当前主库
- 恢复前先转移当前现场

所以不要手工 `cp` 覆盖数据库，除非你正在做极端低层排障并且明白后果。

## 7. 发布 / 打包问题处理顺序

### 7.1 包能不能产出

先跑：

```bash
./scripts/build-app.sh --output-dir /tmp/promptpanel-build
```

确认：

- `/tmp/promptpanel-build/PromptPanel.app` 存在
- `Contents/MacOS/PromptPanel` 存在
- `Contents/Resources` 有 bundle
- `Contents/Frameworks` 有 `Sparkle.framework`

### 7.2 签名是否有效

```bash
codesign --verify --deep --strict --verbose=2 /tmp/promptpanel-build/PromptPanel.app
```

### 7.3 整体发布前检查

```bash
./scripts/release-readiness.sh --output-dir /tmp/promptpanel-ready-check
```

如果这个命令失败，先修它报的第一类问题，不要手动跳过。

当前发布检查还会做一轮隔离恢复演练：

1. 用隔离 `AppSupport` 启动打包后的 `.app`
2. 等待应用创建 `promptpanel.db` 和启动备份
3. 校验启动数据库 integrity
4. 停掉隔离实例
5. 调用 `restore-backup.sh` 把该备份恢复到隔离目标目录
6. 再校验恢复后的数据库 integrity

这一步的意义是确认“能启动”和“能备份”之外，真正能在上线事故时恢复。失败时不要只看应用主流程，要优先检查备份文件、恢复脚本、schema 校验和权限。

### 7.4 公证问题

先确认：

- 已使用 Developer ID Application 签名
- `notarytool` 可用
- keychain profile 正确
- 提交的是签过名的 archive

再执行：

```bash
./scripts/notarize-app.sh \
  --app-path /absolute/path/to/PromptPanel.app \
  --archive-path /absolute/path/to/PromptPanel.zip \
  --keychain-profile AC_NOTARY
```

## 8. 兼容性回归建议

每次修自动粘贴、焦点或热键问题，都至少补以下样本：

- Chrome 普通输入框
- ChatGPT Web
- Claude Web
- VS Code 或 Cursor
- 一个 IM 输入框

记录要求：

- 呼出是否成功
- 搜索框是否拿到焦点
- 目标应用是否恢复前台
- 自动粘贴是否成功
- 失败原因是什么
- 是否能稳定复制兜底

统一记到 [兼容性回归记录](./兼容性回归记录.md)。

## 9. 首次接手或事故后的建议动作

### 首次接手

1. 跑 `scripts/check-docs.sh`、构建和发布前检查
2. 读运行健康
3. 做一次真实执行
4. 做一次手动备份
5. 确认当前是否缺完整 Xcode / 证书 / 公证条件

### 事故后

1. 先收集证据，不要立刻做破坏性操作
2. 优先保留数据库、恢复目录和错误产物
3. 先用现有脚本和运行健康界面判断问题层级
4. 必要时再进入代码修复

## 10. 处理问题时最容易犯的错

- 把“数据库损坏”和“migration 写坏”混为一谈
- 把“终端直跑二进制”的现象当成最终发布包结论
- 因为一台机器缺 `xctest` 就误判成项目测试坏了
- 没看执行日志字段，就把所有自动粘贴问题都归因给权限
- 手工覆盖数据库，绕过现有恢复脚本和保护逻辑
- 只看代码，不看主界面的“运行健康”和“最近执行记录”
- 改了代码、脚本或 UI 基准后忘记同步 [文档与代码同步矩阵](./文档与代码同步矩阵.md) 要求的文档
