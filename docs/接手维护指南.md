# PromptPanel 接手维护指南

更新时间：`2026-05-07`

本文面向新的维护者，目标是让接手人第一天就能把项目跑起来、判断当前风险、知道改哪里、怎么验证、怎么发布和怎么回滚。

## 1. 项目一句话

PromptPanel 是一个本地优先的 macOS 常驻桌面应用。核心价值不是主窗口 CRUD，而是：

```text
全局热键 -> 快捷面板聚焦 -> 搜索词条 -> 先写剪贴板 -> 尝试恢复目标应用并自动粘贴 -> 失败时复制兜底并记录日志
```

正式运行态只有 macOS GUI 会话里的 `.app`。容器和普通服务器不能运行产品本体，只能做 CI、静态产物托管或 Sparkle appcast 分发。

## 2. 第一小时必须确认

在仓库根目录执行：

```bash
git status --short
swift --version
xcode-select -p
xcrun --find xctest
swift build
```

判断规则：

- `swift build` 不通过时，先修构建，不要继续排 UI 或发布。
- `xcrun --find xctest` 失败时，说明当前机器缺少完整 Xcode 测试运行器；可以做构建和本地 QA，但不能声明完整 XCTest 已执行。
- 当前 `docs/` 已随仓库版本化，交接时要确认文档变更和代码变更在同一批交付里保持一致。
- 当前仓库可能已有未提交业务改动，先读 `git diff --stat`，不要为了整理文档回滚别人的代码。

## 5. 日常开发闭环

普通代码改动建议按这个顺序：

```bash
swift build
swift test
```

如果本机没有 `xctest`，用下面命令只能证明测试包可构建，不能当成真实测试通过：

```bash
swift build --build-tests
./scripts/release-readiness.sh --allow-build-only-tests --output-dir /tmp/promptpanel-ready-check
```

UI 或面板改动还要补：

```bash
./scripts/build-app.sh --debug --no-archive --output-dir /tmp/promptpanel-ui-qa
open /tmp/promptpanel-ui-qa/PromptPanel.app
```

需要让 Computer Use 直接看到窗口时，用：

```bash
./scripts/launch-computer-use.sh --surface library
./scripts/launch-computer-use.sh --surface settings
./scripts/launch-computer-use.sh --surface panel
```

## 6. 发布与安装闭环

本地 QA 包：

```bash
./scripts/build-app.sh --output-dir /tmp/promptpanel-build
codesign --verify --deep --strict --verbose=2 /tmp/promptpanel-build/PromptPanel.app
```

发布前统一入口：

```bash
./scripts/release-readiness.sh --output-dir /tmp/promptpanel-ready-check
```

对外分发必须满足：

- 使用完整 Xcode 跑真实 `swift test`，不能用 build-only 降级结果。
- 使用 Developer ID Application 签名。
- 运行 `scripts/notarize-app.sh` 完成 notarization、staple 和 Gatekeeper 复核。
- 配置 Sparkle 时必须同时有 appcast feed URL 和 EdDSA 公钥。
- 至少完成一轮真实目标应用兼容性样本，不只看终端或隔离启动。

## 7. 排障闭环

先判断问题类型：

- 构建 / 测试问题：看 toolchain、SwiftPM、`xctest`。
- 热键 / 面板 / 焦点问题：看 `PanelService`、`PanelOpenTracker`、运行日志和真实 `.app` 形态。
- 自动粘贴问题：先看执行日志里的 `failure_reason`，不要直接归因给权限。
- 数据库问题：区分损坏库自动隔离和 migration 失败保留现场。
- 发布问题：先跑 `release-readiness.sh`，再看签名、公证、恢复演练。

现场证据优先级：

1. 应用内“运行健康”和“最近执行记录”。
2. `~/Library/Application Support/PromptPanel`、`Backups/`、`Recovery/`。
3. `~/Library/Logs/PromptPanel`。
4. 打包脚本和 release-readiness 输出。
5. 真实目标应用样本记录。

## 8. 扩展规则

新增功能时优先沿用现有结构：

- 新数据字段：加 migration，更新模型、仓库、测试、配置文档。
- 新业务动作：放在 Service / Repository / ViewModel，不把逻辑塞进 View。
- 新系统能力：先放 `Integrations/` 做薄封装，再由 Service 编排。
- 新 UI：先改 `frontend-draft/`，再改 SwiftUI/AppKit。
- 新发布能力：优先改脚本和 CI，不塞进主应用运行时。

不要破坏的边界：

- 剪贴板写入是硬保证，自动粘贴只是增强。
- 执行日志只记录诊断元数据，不记录词条正文。
- migration 失败不能自动擦库重建。
- 正式运行态不是容器、不是服务器、不是无 GUI 后台进程。
